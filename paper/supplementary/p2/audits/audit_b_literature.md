# Audit B: Literature Pitfalls for Evaluating Persistent Memory LMs

This memo focuses on one question: how can an evaluation make a recurrent or persistent-memory model appear to remember when the answer is actually available through another path? The newly found D4v2 category-cue leak is a textbook version of this problem: the callback template exposes the answer's semantic class ("favorite COLOR"), letting the backbone learn `P(item | category)` without using the persistent memory state `M_c`. The literature does not use one shared term for this failure, but the same concern appears under "shortcut learning," "literal matching," "benchmark contamination," "spurious cues," "needle-in-a-haystack saturation," "closed-book baselines," and "memory ablations."

## §1 Confirmed pitfalls

**Surface-form and literal-match leakage in long-context retrieval.** NoLiMa argues that many long-context benchmarks, including synthetic NIAH variants and downstream QA-style tasks, contain literal matches between the query and the evidence. Those matches let attention locate candidate evidence by surface overlap rather than by the intended associative or reasoning operation. NoLiMa's fix is to minimize lexical overlap between the question and the needle and to filter haystacks for misleading words or semantically similar false answers. Its empirical result is directly relevant: models that do well on ordinary NIAH drop sharply when literal matches are removed, showing that saturated NIAH performance can be a property of the template rather than robust memory use (NoLiMa, arXiv:2502.05167).

**Needle-in-a-haystack saturation and weak task complexity.** RULER was motivated by the observation that vanilla NIAH is "only a superficial form of long-context understanding." It expands retrieval beyond one passkey into multiple needle types, multiple keys, multiple values, multi-query recall, variable tracking, aggregation, and QA with distractor paragraphs. RULER's important design lesson is that a memory benchmark should vary the target type and the number of hard distractors. Models that score nearly perfectly on vanilla passkey/NIAH still fail when the value is a UUID, when multiple needles share nearby structure, or when irrelevant needles fill the context (RULER, arXiv:2404.06654).

**Position and recency/primacy bias.** Lost in the Middle shows that long-context models often use information best when it is at the beginning or end of the prompt and much worse when it appears in the middle. The paper uses two controlled tasks: multi-document QA with exactly one answer-bearing document among distractors, and key-value retrieval where keys and values are randomly generated UUIDs. This matters for persistent-memory evaluation because fixed introduction/callback positions can become a second shortcut. Even if no category cue exists, a model may exploit position, fixed phrasing, or recency-like structure unless fact positions and session order are randomized and separately stress-tested (Liu et al., arXiv:2307.03172).

**Closed-book and no-context baselines expose parametric priors.** LongBench explicitly evaluates whether models can answer questions without the long context, because public datasets may have been memorized during pretraining. This is the exact diagnostic D4v2 needed: run the callback session alone, or run with memory zeroed/detached, and measure the residual answer CE. If the no-memory/no-context baseline is strong, the benchmark is not isolating memory. LongBench also notes that retrieval can compensate for weak long-context modeling but is not automatically a shortcut to deep long-context understanding; that distinction should be preserved in any memory benchmark (LongBench, arXiv:2308.14508).

**Distractor realism and evidence filtering.** BABILong hides bAbI-style facts inside PG19 book text to make models distinguish task facts from irrelevant natural language. Its authors note that generated benchmarks reduce contamination risk and can scale to arbitrary lengths, but they also show that simple RAG struggles when order and fact chaining matter. The key pitfall is that distractors must be hard enough to prevent template matching, while the answer identity itself must not be predictable from the question. BABILong's use of generated facts plus large natural distractors is a useful middle ground, but the bAbI sentence distribution is still visibly different from PG19, which the authors acknowledge helps models filter task sentences (BABILong, arXiv:2406.10149).

**Benchmark contamination and memorized public data.** ZeroSCROLLS and LongBench both respond to the risk that public train sets have leaked into modern LMs. ZeroSCROLLS uses zero-shot evaluation over held-out validation/test-style data rather than releasing training sets for the benchmark. LongBench samples from test sets and includes experiments that remove the long context. The memory-evaluation analogue is to ensure that answer tokens and bindings are generated per run, not drawn from a stable finite list that a jointly trained backbone can internalize.

**Memory readout can be correct while final decoding still fails.** LongMemEval separates long-term memory into indexing, retrieval, and reading. It finds that even with good recall, using retrieved memories is non-trivial; structured reading prompts and Chain-of-Note improve answer accuracy. This cautions against treating end-to-end CE as a pure memory metric. A persistent-memory system should report at least three quantities: whether the relevant fact was written, whether it was retrieved/read, and whether the decoder used it rather than a prior (LongMemEval, arXiv:2410.10813).

**Shortcut learning is the default under ERM when shortcuts are predictive.** Geirhos et al. define shortcut learning as reliance on decision rules that perform well on standard benchmarks but fail under shifted or stress-test conditions. The D4v2 category-cue leak fits this definition: the shortcut is statistically valid on the evaluation distribution and therefore rational for the model to learn. The lesson is that the benchmark distribution itself must make every non-memory rule uninformative; hoping that an end-to-end model "chooses" the memory route is not a reliable experimental design (Geirhos et al., arXiv:2004.07780).

## §2 How leading papers design memory-only tasks

| Work | Intended memory behavior | Anti-shortcut design choices | Remaining caveat for Memory Residuals |
|---|---|---|---|
| Memorizing Transformers, Wu et al. 2022 (arXiv:2203.08913) | Retrieve internal representations from a non-differentiable kNN memory of recent tokens. | Evaluates domains where recently introduced definitions matter, especially code/math/formal theorem text, and shows gains with larger memory. Uses a memory-size ablation to test marginal value. | Mostly natural LM perplexity, so improvements can mix true memory with domain repetition and local distributional regularities. For a synthetic invariant, add no-memory CE and randomized binding tests. |
| Recurrent Memory Transformer, Bulatov et al. 2022 (arXiv:2207.06881) | Pass information across segments via special memory tokens. | Uses copy, reverse, associative retrieval, and quadratic-equation tasks split across segments. These algorithmic tasks require global information beyond the current segment, and baselines include Transformer and Transformer-XL. | Copy/reverse can still be solved by learning task algorithms when training/test distributions are narrow. For callback facts, use unseen per-example bindings and held-out answer tokens. |
| Associative RMT / ARMT, 2024 | Add associative memory to RMT for long-range retrieval. | Evaluated on long associative retrieval and BABILong-style tasks, including very long contexts where relevant facts are separated by millions of tokens. | Strong evidence for recurrence under extreme length, but fine-tuned synthetic tasks can still reward task-specific parsing unless the answer bindings are fresh and the no-memory decoder is at chance. |
| AutoCompressors, Chevalier et al. 2023 (arXiv:2305.14788) | Compress prior segments into learned summary vectors used as soft prompts. | Trains with randomly segmented inputs, summary accumulation, and ablates plain text vs summary vectors. Evaluates compressed in-context demonstrations and retrieval-augmented LM/reranking. | It optimizes improved LM likelihood, not a strict memory-only invariant. Summary vectors may encode broad priors unless tested with randomized, unguessable bindings. |
| TRIME, Zhong et al. 2022 (arXiv:2205.12674) | Train LMs to use local, long-term, and external memory representations. | Uses in-batch contextualized representations as positive/negative memory candidates; batches consecutive same-document segments for long-term memory and high lexical-overlap other documents for external memory. Compares training/testing memory conditions. | Strong for non-parametric memory training, but standard LM perplexity can improve through lexical recurrence rather than persistent per-entity state. |
| Block-Recurrent Transformer, Hutchins et al. 2022 | Maintain block-level recurrent state vectors across long documents. | Compares against strong sliding-window/Transformer-XL-style baselines on PG19, arXiv, and GitHub; recurrent state has fixed capacity and gates. | Natural-document perplexity measures usefulness of recurrence, not proof that a callback answer is impossible without state. |
| LongMemEval, Yu et al. 2024/2025 (arXiv:2410.10813) | Evaluate chat-assistant long-term memory over many sessions. | Manually curated questions, evidence statements embedded indirectly in task-oriented sessions, timestamped histories, knowledge-update questions, multi-session reasoning, and abstention for false-premise questions. Tests memory systems at indexing/retrieval/reading stages. | It is a QA benchmark, so models may exploit user-profile priors unless closed-book/no-history baselines are reported per question type. |
| RULER, Hsieh et al. 2024 (arXiv:2404.06654) | Stress long-context retrieval and beyond-retrieval behavior. | Uses random words, numbers, UUIDs, multiple keys/values/queries, hard distractor needles, variable tracking, aggregation from sampled word distributions, and QA with random distractor paragraphs. | RULER is long-context, not persistent-memory, but its UUID and hard-distractor recipes are directly transferable. |
| BABILong, Kuratov et al. 2024 (arXiv:2406.10149) | Require reasoning over facts scattered through arbitrarily long contexts. | Generates bAbI facts, embeds them in PG19 distractor text, includes fact chaining, induction, deduction, counting, lists/sets, and reports RAG/fine-tuning baselines. Generated data reduces contamination. | Fact sentences can be distributionally separable from book text; a callback-only setup must remove category priors too. |
| ZeroSCROLLS, Shaham/Ivgi et al. 2023 (arXiv:2305.14196) | Zero-shot long-text understanding. | Provides test/validation-style evaluation without train sets, includes aggregation/fusion tasks where naive baselines remain strong, and avoids benchmark-specific supervised fitting. | Real datasets are harder to prove memory-only; use it as a contamination-control example, not as a template for synthetic callback invariants. |
| LongBench / LongBench-v2, Bai et al. 2023/2025 | Broad long-context understanding and reasoning. | Uses test splits, multi-hop datasets designed/paraphrased to avoid shortcuts, distractor documents, and no-context/memorization checks. LongBench-v2 emphasizes difficult multiple-choice questions requiring comprehension beyond extraction. | Multiple-choice format can introduce answer-prior artifacts; for memory CE, prefer open vocabulary randomized answer tokens. |

## §3 Joint-training pathology: what's the consensus?

The closest consensus is not specific to memory residuals, but it is strong: when multiple pathways can solve a training objective, gradient descent tends to allocate credit to the easiest, most predictive path. If the backbone can solve a callback from template/category priors, it will do so; the memory module receives weak or noisy marginal gradient and can become unused even when architecturally present.

This is the same family as shortcut learning: the model optimizes the proxy distribution, not the researcher's intended causal mechanism. It is also analogous to MoE expert collapse, where learned routing can concentrate traffic in a small subset of experts unless load-balancing or routing constraints preserve specialization. Recent MoE work still treats collapse as an active failure mode: without balancing, expert utilization can shrink during training; with overly strong balancing, performance can degrade. The transferable lesson is that a memory path needs explicit pressure or experimental isolation if the base LM path is sufficient.

RIMs make the complementary point positively. Recurrent Independent Mechanisms use sparse attention/communication and competition so modules specialize and activate only when relevant. That design reflects an assumption that unconstrained dense computation does not naturally decompose into clean mechanisms. For Memory Residuals, depth-routed attention residuals and memory blocks should therefore be audited with usage metrics, path ablations, and interventions, not only end-to-end loss.

Differentiable memory systems add another warning. Neural Turing Machines and Differentiable Neural Computers demonstrated algorithmic memory tasks such as copy, repeat-copy, and associative recall, but they were also known to be difficult to train because many interacting components must learn addressing, writing, and reading jointly. If a simpler recurrent/controller strategy solves the training distribution, external memory need not learn the intended algorithm. In a modern LM, the "simpler controller strategy" can be a very strong parametric prior.

The practical conclusion: joint training is acceptable only if the corpus makes the no-memory channel information-theoretically unhelpful and if the paper reports memory-disabled, memory-shuffled, and backbone-only controls. Otherwise a null marginal memory contribution is not surprising; it is the expected outcome under shortcut learning.

## §4 Negative-result papers worth citing

**RULER vs vanilla NIAH.** RULER is a clean negative result against saturated passkey/NIAH claims. Many models that look excellent on vanilla NIAH degrade under varied needles, hard distractors, multi-hop tracing, and aggregation. Cite this when arguing that "can retrieve a template passkey" is not evidence of robust memory.

**NoLiMa vs literal matching.** NoLiMa is the closest citation to the D4v2 leak. It shows that literal overlap between query and evidence is a confound and that performance falls when the benchmark removes that overlap. The D4v2 category-cue leak is a semantic-prior version of the same phenomenon: the callback text identifies an answer subset even without the memory.

**Lost in the Middle.** This is a negative result for long-context claims: extending the context window does not imply robust use of arbitrary positions. It motivates position randomization and per-position reporting for introduced facts and callbacks.

**Stronger Baselines for RAG with Long-Context LMs.** Laitenberger et al. show that a simple DOS RAG baseline, preserving original document order, can match or outperform more complex multi-stage RAG pipelines such as ReadAgent and RAPTOR under matched token budgets (arXiv:2506.03989). This is useful rhetoric for memory systems: every added memory mechanism should beat the simplest strong baseline under the same information budget.

**BABILong RAG result.** BABILong reports that RAG is not enough for its order-sensitive/fact-chaining tasks, while fine-tuned recurrent memory models can solve much longer sequences. This is a "honest" mixed result: simple retrieval can be strong on single facts but fails when the benchmark makes order and reasoning necessary.

**LongBench no-context memorization checks.** LongBench explicitly measures performance when the long context is withheld. That is a direct precedent for callback-only and memory-zeroed controls. Any memory benchmark should report this number as prominently as the memory-enabled score.

## §5 Concrete recommendations for the v16 corpus

1. **Make answer identity independent of every callback token except through `M_c`.** For each chain/session, draw the answer from a per-example random pool. The callback may ask "What is my favorite item?" but it must not reveal "color," "animal," "city," or any other category that changes the posterior over the answer. If categories are part of the task, balance them so `P(answer | category)` is uniform over a very large pool and test callback-only CE is at the random baseline.

2. **Use nonce answer tokens or held-out token strings.** The strongest proof is to use synthetic identifiers such as `zxqv_1847`, UUID-like strings, or newly added tokenizer tokens whose assignment is generated after model pretraining and split from training. If the answer token has no semantic prior, the backbone cannot infer it from the callback. If using existing tokens, sample uniformly from a large list and ensure every item appears under every category equally often.

3. **Report a no-memory lower bound for every dataset version.** Before training memory, train/evaluate the backbone with memory zeroed, memory detached, memory shuffled across chains, and callback-only input. The dataset should not be accepted unless these controls are near the information-theoretic baseline. For CE, include the expected random CE over the candidate pool.

4. **Add memory-shuffle and session-permutation controls.** A model that truly uses `M_c` should fail when memories are swapped between chains, when the write session is replaced by another chain's memory, or when answer labels are permuted after writing. It should be invariant to irrelevant distractor-session order except where temporal update tasks intentionally require order.

5. **Separate write, retrieve, and decode metrics.** Following LongMemEval, add probes that ask whether the fact was written into slots, whether the reader attends to the relevant slot, and whether the output logit changes when the relevant slot is ablated. End-to-end CE alone cannot distinguish "memory not written" from "memory written but decoder ignored it."

6. **Include hard distractors with same surface form.** For each target fact, include distractor facts using the same template and same category but different chains/entities. This prevents the model from using template position or category alone. RULER's multi-key/multi-value NIAH and BABILong's fact distractors are good precedents.

7. **Randomize templates, positions, and lexical forms.** Use multiple paraphrases for introduction and callback, randomize fact position within the session, and ensure train/test template splits. A held-out template split should remain solvable if memory contains the binding and the callback specifies the key, but not if the answer is inferred from wording.

8. **Add update and abstention tasks.** LongMemEval's knowledge-update and false-premise questions are valuable because they catch stale memory and over-eager priors. For v16, include "my favorite changed from X to Y" and "what is my favorite Z?" where Z was never introduced. Correct behavior should be latest-value recall or "unknown," not the global prior.

9. **Use a balanced factorial generator and publish leakage audits.** For every generated corpus, compute empirical `P(answer)`, `P(answer | category)`, `P(answer | callback_template)`, `P(answer | position)`, and callback-only CE with an n-gram/logistic baseline and the frozen LM. Treat any non-random result as a failed corpus, not as a model result.

10. **Make the main claim conditional on causal interventions.** The paper's invariant should be stated operationally: performance must drop to chance when `M_c` is zeroed/shuffled, remain high when session KV caches are unavailable, and improve monotonically with correct memory capacity or retrieval quality. This is much stronger than "joint model beats backbone."

For the immediate v16 redesign, the cleanest recipe is: per-chain random nonce answer tokens; callback with no category word; balanced key templates; distractor chains with identical templates; memory-zeroed, memory-shuffled, and callback-only baselines reported in the main table. This makes the desired invariant nearly provable: the callback contains the query key but zero information about the answer value, so the only available source for the value is the persistent memory state.
